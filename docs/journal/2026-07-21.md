# 2026-07-21

## Night engineer: daily brief trend buckets by IST, not UTC

**What.** Fixed a real correctness bug in the morning brief's "Last 7 days, day by
day" section. `weeklyTrend()` in `apps/web/lib/ai/daily-brief.ts` grouped screen
time and completed tasks with `to_char(ts, 'YYYY-MM-DD')`. Those columns are
`timestamptz`, so `to_char` renders them in the DB session timezone (UTC on Neon) —
while the whole rest of the app defines a "day" as IST (`istToday()`, `weekMonday`,
`snapshotNetWorth`, health/routine payload days all use `Asia/Kolkata`). Changed both
buckets to `to_char(ts at time zone 'Asia/Kolkata', 'YYYY-MM-DD')`.

**Why.** IST is UTC+5:30, so anything between 00:00 and 05:30 IST was being bucketed
into the *previous* UTC day. That means the coaching data Vivy reads was wrong: a
late-night YouTube session or a task finished at 2 AM showed up under yesterday, and
some days looked emptier than they were. The brief is the Jarvis moment; its numbers
have to be honest. This was the lone UTC-day query left in the brief — now it matches
everything else.

**How verified.** `npm run build` → `✓ Compiled successfully` and `Finished
TypeScript` clean. The build then stops at page-data collection with `No database
connection string was provided to neon()` — that is purely this cloud env having no
`DATABASE_URL`; the parent commit fails at the exact same step with no change from me,
so it is pre-existing and environmental, not from this edit. Could not run against
real data (no DB here), but the change is a self-contained SQL timezone conversion on
`timestamptz` columns, which is well-defined.

**Priority path.** FEEDBACK.md empty, no `[!]` bug lines, no obviously-small unticked
tracker task that was safe tonight → picked a code-quality bug I found: this UTC/IST
day-bucketing mismatch. One change, ~4 lines of SQL plus a comment.

**What the human should check (Vercel preview).** Open the preview, look at a
generated brief's "Last 7 days, day by day" list, and confirm the per-day screen-time
and task-completion numbers line up with your sense of which IST calendar day the
activity happened on — especially anything you did after midnight. Nothing else in the
UI changes.
