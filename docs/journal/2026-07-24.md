## Night engineer: input hardening on the one pipe (`/api/events`)

**Context.** No unhandled feedback (FEEDBACK.md queue empty, no DATABASE_URL in the
cloud env so no live feedback reads), no `[!]` bug lines, and the open `[ ]` tracker
tasks are all user-action steps or large multi-part features. So tonight was a
code-quality pick: the smallest defensive fix with real value.

**What.** Hardened request parsing in `apps/web/app/api/events/route.ts` — the single
pipe every ingestor (browser extension, screen agent, finance, recorder) POSTs to.
Three thin guards, ~15 lines:
- **POST:** reject an event whose `ts` is a malformed date with `400 invalid ts`
  *before* the batch insert. Previously `new Date(e.ts)` produced an `Invalid Date`
  that would fail the whole insert — one bad timestamp in a batch of 500 took the
  entire batch down with a 500.
- **GET `since`:** validate the date; a junk `since` now returns `400 invalid since`
  instead of building a `gte(events.ts, Invalid Date)` condition (broken/empty query).
- **GET `limit`:** clamp to 1–500 with a fallback default of 100. `limit=abc` used to
  compute `Math.min(NaN, 500)` → `.limit(NaN)`.

**Why.** This is the architecture's one choke point — "everything is an event into one
timeline." A single malformed payload from any thin-client ingestor could poison an
ingest or a query. The guards fail fast with a clear message and never touch the happy
path. No schema, no auth, no proxy, no deps — pure route-level validation.

**How verified.** `npm install` then `npm run build`: compiled clean, TypeScript
passed. Full build (page-data collection) needs a connection string at module load, so
ran it once with a throwaway `DATABASE_URL=postgresql://u:p@localhost/db` (no real DB,
nothing written) — all routes built green. No app run against real data (can't from
here), but the change is parse-only and self-contained.

**What the human should check (Vercel preview):**
- Normal ingest still works: `POST /api/events` with a valid event → `{ok:true}`.
- `POST` with `ts:"not-a-date"` → `400 invalid ts (use ISO 8601)` (no row inserted).
- `GET /api/events?since=garbage` → `400`; `GET /api/events?limit=abc` → returns up to
  100 rows (default), not an error.
- The browser extension / screen agent still ingest normally (they send ISO or no `ts`).
