# Journal: 2026-07-19

## Session: night engineer — planner load-error state

**Did:**
- Hardened the /plan page (SPEC-0010 nightly planner) against a stuck loading state.
  Before: `Planner` fetched `/api/plan` on mount, and on any failure ran
  `.catch(() => setCtx(null))`. Since the render guard is `if (!ctx) return "Loading
  tomorrow…"`, a failed context fetch left the page stuck on that spinner forever with
  no way out. Worse, a non-OK JSON error body was parsed and set as `ctx`, so an error
  response rendered as if it were valid plan context.
- Fix (only `app/plan/planner.tsx`): pulled the fetch into a `load()` function, added
  an `res.ok` check before parsing, and a `loadErr` state. On failure the page now shows
  a simple-English message ("Couldn't load tomorrow. Check your connection and try
  again.") in `text-rose` with a "Try again" button that re-runs `load()`. Loading and
  error are now distinct states.

**Why:**
- No unhandled feedback (docs/FEEDBACK.md queue empty), no `[!]` bugs in TRACKER, and the
  open `[ ]` tracker tasks are all user/device/OAuth steps or risky. This was the
  smallest genuinely-valuable, safe item: a real dead-end a user can hit (planner API
  500 / offline) with a clean, contained frontend fix. Diff ~30 lines, one file.

**How verified:**
- `npm run build` (with a placeholder DATABASE_URL — neon() throws at module load
  without one; the build makes no real connection): "✓ Compiled successfully",
  TypeScript finished clean, 37/37 pages generated, exit 0. Couldn't exercise the live
  error path (no app run against real data), but the logic is straightforward and
  type-checked.
