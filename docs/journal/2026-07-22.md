## Night engineer: harden GET query params (no more 500 on bad input)

**Why.** No unhandled feedback (FEEDBACK.md queue empty, no DB access this run), no
open `[!]` bugs, no small ticket I could safely close without the user/browser. So I
picked a code-quality item I found: a duplicated robustness gap.

**What.** `GET /api/events` and `GET /api/transactions` parsed their query params with
`Number(...)` and passed the result straight into Drizzle. A malformed value slipped
through as `NaN` or `Invalid Date`:
- `?limit=abc` → `Math.min(Number('abc'), 500)` = `NaN` → `.limit(NaN)` → Postgres
  rejects it → 500.
- `?since=garbage` → `new Date('garbage')` = Invalid Date → bad SQL → 500.
- `?days=abc` (transactions) → same `NaN` path.

Added a tiny shared helper `apps/web/lib/query.ts`:
- `intParam(raw, {fallback, min, max})` — missing/non-numeric → fallback, else floored
  and clamped to `[min, max]` (also fixes `?limit=0` / negative / huge).
- `dateParam(raw)` — missing/unparseable → `null` (the `since` filter is simply
  dropped rather than crashing).

Wired both routes to use it. Behaviour for valid input is unchanged (same defaults:
limit 100/max 500, days 31/max 365).

**How verified.** `npm run build` passes clean — TypeScript finished with no errors,
all routes compiled (build needs a `DATABASE_URL` string present at page-data
collection because `neon()` instantiates at module load; used a throwaway
`postgresql://…` value — no DB was touched, and the affected routes are
`force-dynamic` so nothing queried at build). Could not exercise a live malformed
request (no app run against real data this run).

**What the human should check on the Vercel preview.** Hit
`/api/events?limit=abc`, `/api/events?since=nonsense`, and
`/api/transactions?days=-5` — each should return a normal 200 with sensible defaults
instead of a 500. Then confirm a normal load of `/finance` and `/browsing` still shows
the same data as before (valid-input path is untouched).
