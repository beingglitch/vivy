## Night engineer: harden the Event API (the one pipe)

No feedback in the queue (docs/FEEDBACK.md empty, no DATABASE_URL this run so no DB
feedback read), no `[!]` bug lines, and the open `[ ]` tracker items are either big
risky features (Gmail/SMS/bank ingestors, Whisper) or "built, needs a real browser
click" that only the human can tick. So tonight was a code-quality fix I found myself.

**What:** made `/api/events` — the single endpoint every ingestor POSTs to — reject
bad input with a clean 400 instead of crashing with a 500.

Three gaps, all in `app/api/events/route.ts`:
- **POST `ts`:** `new Date(e.ts)` on a malformed clock string ("not-a-date") becomes
  an Invalid Date, which Postgres rejects — and because it's one insert, **one bad
  timestamp 500-ed the entire batch of up to 500 events**. A thin client (browser
  extension, screen agent) with a slightly wrong clock string would silently lose a
  whole sync. Now a new `parseTs()` helper validates each event's ts in the same loop
  that already checks source/type; a bad one returns `400 { error: 'ts must be a valid
  date' }`. Missing/empty ts still defaults to now, unchanged.
- **GET `since`:** `?since=garbage` → Invalid Date → 500. Now returns 400.
- **GET `limit`:** `?limit=abc` (NaN), `?limit=0`, or a negative reached `.limit()` and
  errored. Now clamps: a non-finite or non-positive limit falls back to 100, capped 500.

**Why:** the architecture rule is "everything is an event into one timeline" — this is
the most-depended-on endpoint in the whole app, and it's the seam where untrusted thin
clients meet the DB. A single malformed field dropping a 500-event batch is the kind of
quiet data loss that's near-impossible to notice after the fact.

**Verified:** `npm run build` passes clean (TypeScript typechecks; full build green with
a placeholder DATABASE_URL for the page-data collection step — no real DB touched, no
DB writes). Logic traced by hand for each case: valid ts/since/limit unchanged; missing
→ existing defaults; malformed → 400 (POST/GET) or fallback-to-100 (limit).

**Diff:** ~30 lines, one file. Branch `night/harden-events-api` → PR to main.

**What the human should check on the Vercel preview:** the happy path is unchanged, so
existing ingestors keep working. If you want to see the new behavior:
`GET /api/events?limit=abc` returns rows (defaults to 100) instead of erroring;
`GET /api/events?since=xyz` returns `400 { "since must be a valid date" }`;
a `POST` with `"ts":"notadate"` returns `400 { "ts must be a valid date" }` instead of
a 500. Normal extension/agent traffic (ISO strings or epoch numbers) is untouched.
