## Night engineer: header bell shows the unread count, not just a dot

**Context first.** No DB access this run (no `DATABASE_URL` in the cloud env),
`docs/FEEDBACK.md` empty, no open `[!]` bugs. Before picking anything I checked
the open PRs: **six night PRs (#1–#6) are already open and unmerged**, covering
the obvious robustness/error-state ground —
process-events overflow (#1), tasks add-form fail-safe (#2), query-param parsing
(#3), `/plan` load-error state (#4), `/api/events` ts/since/limit hardening (#5),
and negative-money `fmtINRShort` (#6). My first instinct (harden
`GET /api/events` / `transactions` query params) turned out to be an exact
duplicate of PR #3, so I dropped that branch and found something none of the six
touch.

**What changed (tiny, one component).** The header `NotificationsBell` already
fetched the real unread *count* from `/api/notifications?unread=1`, but then
rendered only an anonymous 2px ember dot — so you could tell *something* was
waiting, never *how much*. For a product whose whole Epic 5 is proactive nudges
(morning headline, midday overspend, evening review), "3 waiting" is a stronger,
clearer prompt than "something's there." Now:
- the dot became a small ember count badge (`font-mono`, `text-night` on
  `bg-ember`, matching the design tokens — numbers wear mono), capped at `9+`;
- the `<Link>` carries an accessible label — `"Notifications, 3 unread"` when
  there's a count, plain `"Notifications"` when zero.

No API, data, or logic change — the count was already computed; this only
surfaces it. `app/notifications-bell.tsx` only.

**Why verified.** `npm run build` passes clean — compiled, TypeScript green, all
37 pages generated. (Page-data collection needs a connection string at module
load, so the build was run with a throwaway inline `DATABASE_URL` — no `.env`
file created or changed, nothing connects, no DB touched.) I can't click it in a
real browser without data, so the tracker *tested* box stays `[ ]`.

**What the human should check on the Vercel preview.** With at least one unread
notification, the header bell should show a small ember number badge (e.g. `3`,
or `9+` past nine) instead of the old dot; opening `/notifications` marks all
read and the badge disappears on the next navigation. Everything else is
unchanged.
