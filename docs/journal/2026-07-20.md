## Night engineer: make the morning brief survive a single failing query

**Context.** No unhandled feedback (FEEDBACK.md queue empty, no DB access this run),
no open `[!]` bugs in the tracker, and the unticked `[ ]` tracker tasks are either
big new features (Gmail ingestor, screen-time rollup) or "USER: go click/test it"
steps — nothing small and safe to code there. So I picked one code-quality fix I
found myself.

**What I changed.** `lib/ai/daily-brief.ts` built its whole context in one
`Promise.all([...])` — the two core task reads plus ten enrichment pieces
(browsing stats, memory, weekly trend, learning, spend, structure, goals, tomorrow's
plan, health). `Promise.all` rejects if *any* promise rejects, so a transient failure
in a single peripheral query (say `goalsContext` or `browsingStats`) would throw the
whole function and **write no brief at all**. The brief is the Jarvis moment — the
one output that must show up every morning — so a peripheral hiccup blanking it is
the wrong trade.

Fix: a tiny `safe(promise, fallback)` helper that catches and logs, and wraps each of
the ten enrichment calls with a neutral fallback (`''` for the string contexts,
an `EMPTY_STATS` object for `browsingStats`). The two core task queries stay strict —
a brief with no task data is meaningless, so if those fail we still want to fail. Net
effect: if one enrichment query dies, the brief is generated from whatever else
succeeded instead of not at all, and the error is logged for later.

**Why this one.** Highest value among the safe options: it protects the single most
important daily output, touches one file, changes no behavior on the happy path
(same context, same prompt, same output when nothing fails), and never goes near the
forbidden files (auth/proxy/env/schema/crons/money math).

**Diff:** `apps/web/lib/ai/daily-brief.ts` only (~30 lines: import the `BrowsingStats`
type, add `safe()` + `EMPTY_STATS`, wrap the ten enrichment calls).

**How verified.** `npm run build` — TypeScript compiles clean; a first run with no
`DATABASE_URL` in this env fails at page-data collection (neon() needs a connection
string, unrelated to the diff), so I re-ran with a dummy `DATABASE_URL` and the full
build passed clean (exit 0, all routes emitted). I can't run the app against real
data here, so the degraded path wasn't exercised against a live failure.
