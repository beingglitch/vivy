server into the app — same store, new interface (no separate app, per the

## Addendum — night engineer: add-task form fails safe

Night routine, first code PR. Feedback queue empty, no `[!]` bugs, no DB access
tonight — so I took the smallest real quality item I could see: the /tasks add
form silently swallowed failures.

- `app/tasks/task-list.tsx`: `addTask` had no error handling. On a network blip
  the `await fetch` would throw, so `setBusy(false)` never ran and the **Add
  button stayed disabled until a page reload**. On a non-2xx response it cleared
  your typed text anyway and refreshed — so a failed save looked like a success
  (silent data loss on the first-class feature).
- Fix: wrap the POST in try/catch/finally. Clear the input **only** after the
  save really lands (`res.ok`); on failure keep the words and show a warm rose
  line — "Couldn't save that — check your connection and press Add again."
  `busy` always resets in `finally`, so the button can't lock. `aria-invalid` +
  `role="alert"` so screen readers hear the error; typing again clears it.

**Verified:** `npm run build` passes clean (compile + TypeScript green; page-data
collection needs a DATABASE_URL, so I ran the build with a well-formed placeholder
Neon URL — routes are force-dynamic, nothing connects at build). No DB writes.
Couldn't click it in a real browser (no data), so the tested box stays `[ ]`.

**What the human should check on the preview:** on /tasks, type a task and (with
devtools offline, or by killing the network) press Add — the text should stay and
the rose error should appear; the Add button should still be usable. A normal add
should still clear and refresh as before.
