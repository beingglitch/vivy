# 2026-07-13

## Night engineer — fixed silent event drop in the processing loop (BUG-0001)

**Did:**
- Fixed a real correctness bug in `lib/ai/process-events.ts`. The loop mines only
  the first 25 text events and first 60 video watches per run; every unprocessed
  text/video event *beyond* those caps was being marked `processed: true` with no
  task extraction and no category. Since the loop only re-selects `processed: false`
  rows, those events were lost for good — tasks never proposed, videos stuck
  "not yet classified".
- Root cause: `restIds` (the "analytics, needs no AI" set) was "pending events not
  in the *sliced* text/video arrays", which swept up the over-cap overflow. Fixed by
  defining the rest set by type (`!isText(e) && type !== 'video.watch'`) instead of by
  the capped slices, so overflow now stays pending and the next run handles it.
- Wrote up `docs/bugs/BUG-0001-*`; added the Bugs line + a note on the tracker's
  processing-loop item.

**Why:** No unhandled feedback (queue empty, no DB access this run), no open `[!]`
bugs. Scanning code quality I found this — it's reachable in normal use: `POST
/api/events` takes batches up to 500, so one browser-extension push of >60 watches
drops the excess silently.

**Verified:** `npm install` + `npm run build` clean (compile ✓, TypeScript ✓, full
build ✓ with a placeholder `DATABASE_URL` to get past neon() page-data collection —
no real DB in this env). Regression demonstrated with a standalone script: 30 notes
+ 70 videos + 5 page-visits → OLD marks 20 processed-without-AI incl. 15 unmined
text/video (lost); NEW marks only the 5 real analytics events, leaving 5 text + 10
video overflow pending for the next run.

**Status:** on branch `night/process-events-overflow`; PR to main opened. Never
touched schema/auth/proxy/env/crons/deps.
