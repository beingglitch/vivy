## Harden numeric query-param parsing on the timeline GETs

**What.** Added `apps/web/lib/query.ts` with one helper, `intParam(raw, {fallback,
max, min=1})`, and used it in the two GET routes that read a number off the URL:
- `GET /api/events` — `limit` (fallback 100, max 500)
- `GET /api/transactions` — `days` (fallback 31, max 365)

Both previously used `Math.min(Number(param ?? default), max)`, which is fragile:
- `?limit=abc` → `Number('abc')` is `NaN` → `Math.min(NaN, 500)` is `NaN` →
  Drizzle emits `LIMIT NaN`, a DB error.
- `?limit=-5` → passes straight through → `LIMIT -5`, a DB error.
- `?days=-1` → `since` lands in the *future* → the query silently returns nothing.
- `?limit=12.7` → non-integer limit.

`intParam` returns the fallback for missing / empty / non-numeric input, floors
the value, and clamps it into `[min, max]`, so nothing degenerate ever reaches
the database. Empty (`?limit=`) is treated the same as absent.

**Why.** No user feedback in the queue (no DATABASE_URL in this env, `docs/FEEDBACK.md`
empty), no open `[!]` bugs on the tracker. Next priority is code quality I find
myself: this was a real robustness gap — a duplicated fragile pattern across the
project's core "one pipe" read path and the finance read path. Smallest safe fix
that removes both the bug class and the duplication (~20-line helper, two 1-line
call-site swaps). Nothing risky touched (no money math, no auth, no schema).

**How verified.**
- `npm run build` passes clean (TypeScript + full route build) — had to pass a
  placeholder `DATABASE_URL` because `neon()` constructs at module-eval and this
  env has no DB string; that's an env limitation, not a code error.
- Unit-checked `intParam` edge cases with node: `null→100`, `'abc'→100`, `'-5'→1`,
  `'9999'→500`, `'12.7'→12`, `''→fallback`, `'0'→1`, `days '-1'→1`, `days '999'→365`.
- prettier (`--single-quote --print-width 110`) applied to touched files.

**What the human should check.**
